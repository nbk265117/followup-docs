Description des branches

Branche Description Crée depuis Merge vers
main Code en production, stable, tague - -
develop Branche d'integration, dernières fonctionnalites main release/*
feat/* Developpement d'une fonctionnalite develop develop
release/* Preparation d'une release (tests finaux) develop main + develop
hotfix/* Correction urgente en production main main + develop
tag vX.Y.Z Version taguee pour release en production main -

Regles de nommage

Convention de nommage des branches :

feat/[JIRA-ID]-description-courte
bugfix/[JIRA-ID]-description-courte
hotfix/[JIRA-ID]-description-courte
release/v1.0.0
tag: v1.0.0 (sur main apres merge release)

Exemples :
feat/FUP-123-user-auth
feat/FUP-124-eval-form
bugfix/FUP-125-login-error
hotfix/FUP-126-security-fix
tag: v1.0.0, v1.0.1, v1.1.0

Workflow de developpement

1. Creer une branche feat

git checkout develop && git pull && git checkout -b feat/FUP-XXX-ma-feature

2. Developper et commiter

Commits atomiques avec messages clairs. Format: [FUP-XXX] Description du changement

3. Pousser et creer une Merge Request

git push -u origin feat/FUP-XXX-ma-feature → Creer MR vers develop

4. Code Review

Review obligatoire par au moins 1 developpeur (Tech Lead ou Senior Dev)

5. Merge vers develop

Apres approbation, merge de la MR. La branche feat est supprimee.

CONVENTIONS DE CODE

Backend (Java/Spring)

  • Style Guide : Google Java Style Guide
  • Linter : Checkstyle
  • Nommage classes : PascalCase (UserService, EvaluationController)
  • Nommage methodes : camelCase (getUserById, createEvaluation)
  • Packages : com.boa.followup.[module]

Frontend (Angular/TypeScript)

  • Style Guide : Angular Style Guide (officiel)
  • Linter : ESLint + Angular ESLint
  • Composants : PascalCase (UserCardComponent, EvaluationFormComponent)
  • Services : PascalCase avec suffix "Service" (AuthService, EvaluationService)
  • Fichiers : kebab-case (user-card.component.ts, evaluation.service.ts)
  • Modules : Feature modules (UserModule, EvaluationModule)

Messages de commit

Format : [TYPE] Description courte

Types :
feat: Nouvelle fonctionnalite
fix: Correction de bug
docs: Documentation
style: Formatage (pas de changement de code)
refactor: Refactoring
test: Ajout/modification de tests
chore: Maintenance (deps, config)

Exemple : feat: add user authentication with JWT
